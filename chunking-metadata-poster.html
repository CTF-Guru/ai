RAG CHUNKING

How you split & label content decides what RAG can find — the hidden quality lever, from bad cuts to semantic chunks with metadata

01
What chunking is — the step before retrieval that quietly sets the ceiling

Chunking is how source content is split into retrievable pieces for RAG — and how each piece is labeled with metadata before it ever reaches a vector index.

The model only ever sees the chunks you retrieve. If a chunk cuts a procedure in half, drops its heading, mixes two topics, or carries no source — retrieval has nothing good to find, no matter how strong the model is. Chunking is the data-prep decision that sets the quality ceiling for everything downstream.

02
Bad cuts vs good chunks — same document, split two ways

Same source, two splits. The good lane keeps a section whole, preserves its heading, and tags every chunk — so the right evidence is findable and citable.

03
Metadata to attach to every chunk — the labels that make a chunk filterable & trustworthy
📄 title § section / heading 🔗 source URL / doc ID 👤 owner 🗓️ date #️⃣ version 🔐 access group ⚠️ sensitivity 🗂️ document type

These aren't decoration. Section & title restore the context a cut throws away; source & version power citations and freshness checks; access group & sensitivity let retrieval filter before a private chunk ever reaches the model. No chunk ships untagged.

04
Chunking choices — match the strategy to the content
📏
Fixed-size — N tokens with overlap. Simple & fast; blind to meaning, cuts mid-thought.
🧩
Semantic — split on headings / topic shifts. Keeps a section whole — the default for prose.
🌳
Parent–child — retrieve a small chunk, hand the model its larger parent for context.
▦
Table-aware — keep rows + header together; never slice a table across chunks.
{ }
Code-aware — split on function / class boundaries, not arbitrary line counts.
05
Common failure
⚠️
Poor chunks → poor retrieval — even with a strong model. The chunk is the unit of retrieval; if it's malformed, no reranker or prompt rescues it.
🪓
One size for everything — fixed-size cuts shred tables, code, and procedures. Different content needs different splits.
🏷️
Untagged chunks — no source means no citation; no access group means private docs leak into answers.
06
The takeaway
Design chunks around how users ask questions and how evidence should be cited — not around a round token count. A retrievable, well-labeled chunk is what turns a strong model into a trustworthy answer.
Split on meaning→ Tag every chunk→ Filter on metadata→ Cite the source
07
Where it lives in the AI SDLC
Data Prep→ Build→ Evaluation

Chunking & metadata are a Data Preparation decision — made before you build the index, then carried through Build (the retrieval pipeline reads those tags) and pressure-tested in Evaluation, where bad chunks show up as low retrieval recall long before the answer looks wrong.

Get the split right at ingest — it's the cheapest place to fix retrieval quality.
