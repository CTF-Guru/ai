01
What a handoff contract is — the structured task package one agent gives another

When Agent A asks Agent B to do work, the handoff is everything that crosses the boundary: the goal, the context, the limits, and the shape of the answer expected back. A contract makes that package explicit instead of a hopeful sentence — so the receiver knows exactly what "done" means before it starts.

It is a discipline, not a new protocol: it rides on whatever channel you already use to delegate — an A2A task, a sub-agent call, a tool/function invocation, or a queue message. The medium varies; the fields are the same.

The handoff in motion — a task card collects fields as it moves Planner → Research → Review → Action
03
The eight contract fields — what every handoff should carry

🎯 Objective

The single outcome wanted, in one testable sentence.

🗂️ Context

Facts, inputs & prior decisions the receiver needs.

🚧 Constraints

Hard limits — policy, scope, budget, do-nots.

🧰 Allowed tools

The exact tools/permissions it may use — no more.

📐 Output schema

The shape of the result, so it's machine-checkable.

⏱️ Due / SLA

Deadline or time budget before it must report back.

📊 Confidence

How sure it must be — and how to flag uncertainty.

🚨 Escalation

When to stop and hand back to a human or caller.

Objective + Context + Constraints scope the work; Tools + Schema make it safe and checkable; SLA + Confidence + Escalation keep it accountable.

05
Why vague handoffs fail
🔁
Delegation loops — receiver re-asks "what do you mean?"; work ping-pongs, never finishes.
👯
Duplicate work — no shared objective, so two agents redo the same step.
💥
Unsafe actions — no constraints or escalation rule, so an agent sends, spends, or deletes unprompted.
🌫️
Silent drift — no output schema, so a wrong-shaped result slips downstream unnoticed.

Vague handoffs create loops, duplicate work, and unsafe actions.

06
The contract, on the wire — a handoff is just a structured object the receiver can validate
// Planner → Research : a task contract
{ "objective": "draft reply to refund request #812",
  "context": { "ticket": "#812", "tier": "gold" },
  "constraints": [ "no send without approval" ],
  "allowed_tools": [ "kb.search", "policy.lookup" ],
  "output_schema": "DraftReply",
  "sla_seconds": 120, "min_confidence": 0.8,
  "escalate_if": "refund > 50 USD" }

Because it's structured, the receiver can reject a malformed handoff up front — a missing objective or escalation rule fails validation before any tool runs.

✅
Validate on receive — required fields present & well-typed.
🧾
Echo the contract — return result tagged with the objective it answers.
07
Each role's job in the relay
🧭
Planner — writes the objective, constraints & escalation rule.
🔎
Research — fills context with evidence; stays read-only.
🛡️
Review — checks the draft against constraints + schema.
⚙️
Action — executes only what the contract permits — gated.

The contract is the baton — it travels intact; only its filled fields grow.

08
Don't confuse the layers
🤝
Handoff ≠ transport — the contract is what crosses; A2A / a sub-agent call is how.
🧰
Handoff ≠ tool call — a tool runs a function; a handoff delegates a goal to a reasoner.
📇
Contract ≠ Agent Card — the card advertises capability; the contract is one specific work order.
09
Where it lives in the AI SDLC — a design-time pattern with an eval-time payoff
DESIGN define the contract schema & required fields → ORCHESTRATION agents pass & validate contracts at runtime → EVALUATION score handoff quality; catch loops & drift

A clean contract isn't just tidy — it's testable: you can assert that every handoff carried an objective, stayed in-scope, and escalated when it should. Good handoffs are how you evaluate a multi-agent system at all.

10
The takeaway
Treat agent handoffs like API contracts for reasoning work — explicit inputs, a typed output, and a rule for when to stop. Vague asks scale into chaos; signed work orders scale into teams.
Name the objective→ Attach constraints + schema→ Validate on receive→ Escalate by rule
