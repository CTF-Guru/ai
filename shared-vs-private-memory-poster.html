SHARED vs PRIVATE MEMORY

Who is allowed to remember what, across a team of agents — private · team · global, with a policy gate at every boundary
CTF Farooq Ali
· AI SDLC course · explainer series
v2026.06
01
What this is — memory boundaries in a multi-agent system

Shared memory lets agents collaborate. Boundaries decide what stays private, what is team-scoped, and what is globally governed.

One agent's notes are not automatically everyone's notes. Each piece of memory has a scope — and crossing from one scope to a wider one is a deliberate, policy-checked move, not a side effect of two agents talking.

scope ≠ shared-by-default widen only through a gate narrow is always safe
02
The three memory zones — gates sit on the boundaries, not inside the zones

Allowed at the first gate (a sharable decision moves into team scope) — blocked at the second (it isn't org-approved knowledge yet).

03
Private memory holds

Bound to one user or one agent. Default-deny outward.

👤
User preferences — tone, defaults, personal context.
🧠
Scratch reasoning — intermediate thoughts, dead ends.
📎
Session state — what this agent is mid-doing right now.
04
Team memory holds

Shared across one project's agents & people.

🗳️
Decisions — what was chosen, and why.
✅
Open tasks & status — the shared to-do and progress.
🤝
Handoff summaries — the working context agents pass on.
05
Global memory holds

Org-wide, long-lived, owned & reviewed.

📜
Approved policies — the rules everyone inherits.
📦
Product knowledge — stable facts about the product.
🧭
Standard procedures — the approved way to do things.
06
What the gate checks — controls for moving memory between zones

👑 Owner

Who owns this memory and can promote, edit, or retire it.owner = team-lead

🔑 Access group

Which users / agents the wider scope actually exposes it to.group = project-x

✍️ Consent

Did the user agree this may be shared beyond them?consent = explicit

🏷️ Sensitivity

Label that can block or force redaction before sharing.label = confidential

⏳ Expiration

When this memory must be forgotten or re-reviewed.ttl = 30d

🧾 Audit trail

Who promoted, read, or changed it — and when.who · when · why

A promotion that fails any one check does not cross. Narrowing a scope never needs a gate; widening always does.

07
Worked example — a customer-escalation workflow, one fact in each zone
ZoneWhat it may hold hereWhy it stays in this zone
PrivateThe customer's raw message + the agent's unfiltered draft replyMay contain personal / sensitive details; not yet vetted to share
Team"Escalation #4821 open · owner = Dana · blocked on refund policy"The project's agents need shared status to coordinate the fix
GlobalThe approved refund policy & escalation procedureOrg-owned, reviewed knowledge every agent may safely read

The escalation's status rises to team scope so agents collaborate — the customer's raw details stay private, and only approved policy is read from global.

08
The common failure
🩸
Private leaks into shared — a user preference or sensitive detail lands in team / global context, where other agents and people now see it.
🌊
Blast radius — once promoted, it spreads to everyone in the wider scope, and you can't un-share it.
🕳️
Silent promotion — sharing that happens with no consent, label, or audit becomes a trust and compliance gap.
09
Rule of thumb
private = default → promote on purpose

Start every memory in the narrowest scope. Widen only when there's an owner, a reason, and a gate that passed.

When unsure which zone a memory belongs to — keep it private.

10
The takeaway
Design memory boundaries before scaling to multi-agent. Zones + gates are cheap to add early — and nearly impossible to retrofit once private context has already leaked.
Name the zones→ Gate the boundaries→ Then add agents
11
Where this lives in the AI SDLC
Architecture→ Security→ Governance→ Monitor

Decide the zones in architecture, enforce the gates as a security control, set ownership & retention in governance, and watch promotions in monitoring.
