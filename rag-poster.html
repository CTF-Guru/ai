HOW RAG WORKS

Give the model YOUR facts — retrieve, augment, generate
CTF Farooq Ali

01
The problem — a bare LLM is confidently wrong
🧠
Answers from memory — it pattern-completes from training, not from a fact-store. It can hallucinate — fluent, plausible, wrong.
📅
Frozen at the cutoff — training stopped on a date. It knows nothing newer — today's pricing, this week's policy.
🔒
Blind to YOUR data — your private docs, wiki, tickets, codebase were never in its training set.

The fix isn't a bigger model — it's giving it the right facts to read.

02
What RAG is — read first, then answer

Three verbs. The model looks things up before it speaks instead of answering from memory.

① Retrieve→ ② Augment→ ③ Generate

Fetch the relevant facts → bolt them onto the prompt → let the model write a grounded answer. Knowledge and reasoning, kept separate.

03
The indexing pipeline — offline · prepare your knowledge once

Done ahead of time, not per question. Add or change a doc → re-embed just that doc and it's searchable. The store holds the facts; the model never memorizes them.

04
The query pipeline — online · what happens on every question

Embed the question → find the nearest chunks → paste them into the prompt → the LLM answers from those passages, and can cite its sources.

05
Embeddings & vector search

Text becomes a vector — coordinates in "meaning space". Close vectors = close meaning. You search by meaning, not keywords.

06
Chunking

Split docs into passages with the right size + overlap. The dial everyone gets wrong:

🧱
Too big — noise drowns the answer; the model loses the thread.
🔪
Too small — context gets cut; a fact is split across chunks.

Semantic chunking — cut on meaning (headings, topics), not a blind character count.

07
Reranking

Similarity search is fast but rough. A reranker takes the top-k and re-scores them so the best evidence reaches the model first.

top-20 (rough)→ rerank→ top-5 (best)

A cheap second pass that lifts answer quality a lot — fewer, sharper chunks beat many fuzzy ones.

08
Agentic RAG — retrieval becomes a reasoning loop, not one shot

Put RAG inside an agent loop: the agent rewrites the query, retrieves, checks whether it has enough, and retrieves again until it can answer.

rewrite query→ retrieve→ gap? missing info↺ multi-step retrieve→ answer

One-shot RAG retrieves once and hopes. Agentic RAG keeps pulling threads until the evidence is complete — at the cost of more calls.

09
What nobody tells you — where RAG quietly fails
🎯
Wrong source retrieved — the right answer exists but the search missed it. Garbage in → confident garbage out.
🕰️
Stale embeddings — docs changed, the index didn't. Re-index when content changes.
🔑
Missing access control — retrieval can surface docs a user shouldn't see. Filter by permission first.
📊
Weak reranking — good chunk buried at rank 15 never reaches the model.
🌊
Context overload — too many chunks → "lost in the middle"; the answer drowns.
🧪
Hallucinated synthesis — chunks are right; the model still stitches a wrong claim across them.
♾️
Retrieval-loop explosion — agentic RAG re-queries forever. Cap the steps.
☠️
Context contamination — an untrusted retrieved doc carries an injected instruction. Treat retrieved text as data, not commands.
10

Fine-tuning teaches how to behave, not what's true today. Long-context is simple but pricey and "loses the middle". For current, private, citable facts → reach for RAG.

11
The takeaway
RAG separates the knowledge from the reasoning. The knowledge base holds the facts; the model supplies the explanation. Change a document → instantly current, no retraining.
