01
What a trust boundary is — the line where authority changes hands

A trust boundary is any point where control, identity, data access, or execution authority changes hands. MCP hands the model real capability — so each crossing has to be designed, not assumed.

Control

Who decides what runs next — the user, the app, or the model?

Identity

On whose authority does the action execute downstream?

Data & execution

What can be read, written, or run once the line is crossed?

MCP is an integration pattern, not automatic safety. Permissions, validation, identity and logging still need design.

The boundary map — every arrow is a line where trust is re-checked
03
Five permission levels — classify every tool before you expose it
👁️ READ  ·  look, don't touch ✏️ WRITE  ·  create / update state ⚙️ EXECUTE  ·  run code / commands 🗑️ DELETE  ·  destroy / irreversible 📤 EXTERNAL‑SEND  ·  leaves your perimeter

Risk climbs left → right. Read is cheap to grant; write, execute, delete and external-send each widen the blast radius and should demand stronger controls.

04
The permission gate

Six controls that decide whether a call passes:

🔒
Least privilege — grant the narrowest scope the task needs, nothing more.
✅
Tool allowlists — only named, approved tools are callable.
🙋
Human approval — high-risk actions pause for a sign-off.
🧪
Input/output validation — check arguments and results, both ways.
📋
Audit logging — who/what/when/why for every call.
📦
Sandboxed execution — contain blast radius if a call misbehaves.
05
The gate in motion — read passes, write waits

Risk-tiered, not blanket: low-risk flows; high-risk waits for a human.

07
The common failure — oversized blast radius
💥
One broad MCP server + broad credentials = one oversized blast radius. A single over-scoped server holding admin-grade keys means any mistake — or any injected instruction — can read, write, delete, and exfiltrate across everything it touches.
🪪
Shared service account — every agent acting as the same all-powerful identity erases who actually did what.
🧨
"MCP = safe" assumption — the protocol standardizes the connection; it does not scope, validate, or approve anything for you.
08
Bad pattern vs good pattern

✗ Master key

One server, one admin token, every tool — read, write, delete, send — all ungated.

✓ Scoped keys

Servers split by risk & owner; per-tool scopes; approval on destructive calls; every call logged.

Split MCP servers by risk, ownership, and permission boundary — not convenience.

09
Treat tools like prod APIs
Scoped→ Validated→ Logged→ Revocable

Each MCP tool is a production endpoint: it has a narrow contract, checks its inputs, records its use, and its access can be pulled the instant it's misused.

10
Builder takeaway
Design MCP tools like production APIs: scoped, logged, validated, and revocable. Give the agent a labelled keyring — never the master key.
Scope it→ Validate it→ Gate the risky calls→ Log everything→ Revoke on misuse
11
Where this lives in the AI SDLC
Security Design→ Build→ Release Gate→ Monitor

Decide boundaries and permission levels in design; wire scopes, validation and allowlists in build; verify approvals and audit at the release gate; watch the audit trail in monitoring.
