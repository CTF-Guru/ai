01
What RAG security is — the retrieval layer is an attack surface

RAG security protects the model from malicious, stale, unauthorized, or poisoned content retrieved into the context window.

Retrieval-augmented generation grounds answers in fetched documents — but those documents are untrusted input. Any text the model reads can try to act as an instruction. The core defense is a single discipline: retrieved content is data to cite, not commands to obey.

02
The flow — question + docs in, one poisoned doc quarantined, answer grounded in approved evidence
03
Risk 1 · Indirect prompt injection

Retrieved content contains instructions that try to manipulate the model — text the user never wrote, smuggled in through a fetched page, file, ticket, or email.

04
Risk 2 · Context poisoning

Bad content enters the knowledge base or retrieval index — at ingest, not at query time. Corrupted, attacker-planted, or stale documents become "trusted" sources that the retriever will happily rank and serve.

🧫
Planted — attacker uploads a doc to a scraped/open source.
🕰️
Stale — outdated policy outranks the current one.

05
Controls checklist — layered defense, not one switch
✅
Source allowlists & trust scoring — retrieve only from vetted corpora; rank by source trust.
🔐
Access control before retrieval — filter by the user's permissions, then search.
🧪
Content scanning & policy filters — strip/flag injection patterns at ingest & query.
🪜
Clear instruction hierarchy — system > developer > user > retrieved. Docs sit at the bottom.
🔗
Citation & source traceability — every claim links back to an approved document.
📦
Sandboxed tool actions — least privilege; a doc can never trigger a real side-effect.
🎯
Red-team & eval tests — injection/poisoning suites run in CI before release.
🧹
Index hygiene — provenance, signing, and expiry on every ingested chunk.
06
The instruction hierarchy
System› Developer› User
› Retrieved docs — lowest

A document can inform the answer. It can never outrank the user or the system. Enforce the order; don't assume the model will.

08
The common failure — one assumption sinks the whole system
💥
Treating retrieved content as trusted instructions. The retriever's job is to find relevant text — not safe text. Concatenating docs straight into the prompt hands the attacker a microphone.
🧭
The rule: Retrieved content is evidence, not authority. It answers "what do the sources say?" — never "what should I do?"
09
Prove it with red-team evals
🧨
Injection corpus — seed docs with "ignore instructions" payloads; assert the model refuses.
☣️
Poison drill — plant a false fact in the index; assert it's caught or out-ranked.
🚫
Action gate — assert no document can trigger a tool/side-effect.

Gate the build: evals run in CI, not after launch.

11
The takeaway
RAG security is one rule, enforced everywhere: never let documents become hidden system prompts. Retrieve from trusted sources, rank by provenance, cite every claim — and keep instructions above evidence.
Allowlist sources→ Scan & rank→ Evidence, not authority→ Cite & sandbox→ Red-team in CI
