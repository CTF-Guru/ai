01
What agent identity is — the answer to "on whose authority did this run?"

Agent identity defines which software actor is acting, for which user, with which permissions. When an agent acts on your behalf, identity and authorization can't be implicit — every action needs a traceable owner.

Actor

Which agent — not just "the AI" — executed the call?

On behalf of

Which human or service authorized the work?

With what scope

What is this agent permitted to do, and where does it stop?

02
The identity chain — a token is re-checked at every gate

Identity rides with the request. A known, in-scope token clears the gate, runs, and is logged; an unknown or over-scoped one is denied before it touches a tool.

03
Five concepts to keep straight — from "who asked" to "prove it later"

User identity

The human or service requesting the work.

Agent identity

The autonomous or semi-autonomous actor that executes.

Delegated authority

What the agent may do for the user — and nothing beyond.

Tool authorization

What actions each tool permits, per identity and scope.

Audit trail

Evidence of who / what / when / why — reconstructable after the fact.

04
Bad pattern vs good pattern

✗ Shared admin key

One credential for every agent — broad, permanent, anonymous. Any action is "the AI," accountable to no one.

✓ Scoped identity

Per user / session / agent: scoped, revocable, logged. Pull access the instant it's misused.

A shared service account with excessive permissions is the single most common identity failure.

05
What "an identity" must carry
🎯
Scoped — least privilege per user, session and agent — not a blanket grant.
⏳
Time-bound — short-lived tokens tied to one session, not standing keys.
🔁
Revocable — access can be pulled the instant an agent misbehaves.
📋
Logged — every action attributable to a real, named actor.

No identity, no accountability. If the log can't name the actor and the grant, the action can't be governed.

07
The common failure — an access-control blind spot
🕳️
An agent with unclear identity = an access-control blind spot. If you can't say which actor took an action, on whose authority, you can't scope it, can't revoke it, and can't prove what happened.
🪪
Shared service account — every agent acting as the same over-permissioned identity erases who actually did what.
🗝️
Standing master credential — a long-lived broad key is a permanent liability: one leak exposes everything it can reach.
08
Treat agents like prod principals
Identified→ Scoped→ Revocable→ Logged

Each agent is a first-class actor in your access model: it has a name, a narrow grant, an off switch, and a trail — exactly like a production service account done right.

09
How authority is delegated
User grants→ Agent receives→ Tool checks

The agent acts on behalf of the user — never above them. Its authority is a subset of the user's, narrowed to the task, and the tool re-verifies it at the point of use.

10
Builder takeaway
Give agents identities, not magic keys. Scoped, revocable, logged permissions per user, session and agent — so every action has a real, named owner.
Identify the actor→ Scope the grant→ Gate the action→ Log who did it→ Revoke on misuse
11
Where this lives in the AI SDLC
Security Design→ Build→ Release Gate→ Monitor

Decide the identity model and delegation rules in security design; wire per-agent scoped credentials in build; verify revocation and audit coverage at the release gate; watch the attributed audit trail in monitoring.
