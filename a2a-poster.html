How AI agents discover, talk to, and delegate to each other — the open standard for agent interoperability

01
The problem A2A solves — agents on different stacks can't collaborate

Agents are built on different frameworks and vendors — without a shared language they stay islands. A2A is the common tongue between them.

02
What A2A is — an open, vendor-neutral protocol for agents to talk to agents
AtlassianSalesforceSAP ServiceNowMongoDB+ 45 more

Announced by Google (Apr 2025), then donated to the Linux Foundation so no single vendor owns it. An open standard for agent interoperability.

04
The two roles

🧭 Client agent

Initiates — delegates a task and waits on the result.

⚙️ Remote agent

Receives the task, executes it, returns the output.

Agents are opaque. A remote agent exposes its capabilities, never its internal tools, memory, or state — a built-in security boundary.

05
The Agent Card — how agents discover each other

A small JSON metadata document published at a well-known URL — the agent's public "business card". This is how one agent finds and sizes up another before delegating.

🆔
Identity — name, description, provider, version.
🎯
Skills / capabilities — what it can actually do.
🔗
Endpoint URL + supported modalities (text, files, data).
🔑
Auth requirements — OAuth 2.0 / API keys it expects.
// GET /.well-known/agent.json
{ "name": "Weather Agent",
  "url": "https://wx.example/a2a",
  "version": "1.0.0",
  "skills": [ { "id": "forecast",
    "description": "daily forecast by city" } ],
  "defaultInputModes": ["text"],
  "authentication": { "schemes": ["oauth2"] } }
06
The task lifecycle — long-running & async by design
submitted→ working→ input-required→ completed ✓
terminal states also include failed canceled — a task can pause at input-required to ask the client for more, then resume.

A Task is the unit of work. Because real work takes time, A2A treats every task as a stateful, resumable object — not a single blocking call.

07
Messages, Parts & Artifacts

Message

One turn between agents — carries one or more Parts.

Part

A piece of content.TextPart · FilePart · DataPart

Artifact

The remote agent's output — the result it returns.

DataPart carries structured JSON, so agents trade machine-readable results — not just prose. Messages go in, Artifacts come out.

08
Capability & modality negotiation
🗂️
Read the card first — skills + supported modes tell the client what's possible before it asks.
🔀
Negotiate modalities — text, forms, files, audio/video — agree what each side can send and receive.
📦
Agree content types — so the response arrives in a shape the client can actually use.

Discovery isn't just "are you there" — it's "what can you do, and how should I talk to you".

09
On the wire — JSON-RPC 2.0 over HTTP(S), with streaming
// client agent delegates a task to a remote agent
{ "jsonrpc": "2.0", "id": 7, "method": "message/send",
  "params": { "message": {
    "role": "user",
    "parts": [ { "kind": "text",
      "text": "3-day forecast for Kyoto" } ] } } }
📡
SSE — Server-Sent Events stream live progress as the task runs.
🪝
Push notifications — webhooks call the client back when a long task finishes.
🌐
Plain HTTP(S) — standard web transport, easy to host & secure.

Familiar web plumbing — nothing exotic to deploy.

10
Enterprise-grade by design — what nobody tells you
🔒
Opaque by design — unlike MCP's exposed tools, a remote agent reveals only capabilities. Don't expect to see its internals.
⏳
Task state is yours to manage — async, long-running tasks mean tracking IDs, retries, and timeouts.
🪪
Trust the discovered card — an Agent Card is a claim; verify identity & auth before you delegate sensitive work.
📐
Protocol versioning — agents on different A2A versions must negotiate; capabilities drift over time.
🛡️
Standard web auth — OAuth 2.0 / API keys — reuse the security you already run.
🔭
Observability — trace tasks across agent boundaries; a failure can hide one hop away.
11
The complete picture — an orchestrator delegating to specialists

Each specialist is a separate agent — possibly a different vendor — reachable through its Agent Card. The orchestrator never needs to know how they work inside.

12
The takeaway
A2A is the handshake between agents; MCP is the handshake to tools. Together they turn lone agents into teams — build agents that collaborate.
