01
What "advanced" means — basic RAG is two steps; production RAG is a pipeline

Basic RAG is retrieve → generate. It works in a demo and breaks in production: it grabs the top-k chunks and hopes they're right.

Advanced RAG turns that one hop into a quality pipeline — clean up the question, search more than one way, filter by permission & freshness, rerank for the best evidence, assemble a tidy context packet, ground the answer, attach citations, then evaluate what came out. Each stage is a place to measure and improve.

Basic: retrieve→ generate vs a measured pipeline at every hop
02
The pipeline — question in, grounded + cited answer out

Eight stages, each measurable. The win isn't one clever step — it's the chain of quality gates.

03
Query rewrite

The user's words are rarely the best search words. Rewrite to expand acronyms, add synonyms, split multi-part questions, or resolve "it / that" from chat history.

Search the meaning, not the typo.

04
Hybrid retrieval

Semantic

Vector / dense. Finds meaning, even with different words.

Keyword

BM25 / sparse. Nails exact terms, IDs, codes, names.

Fused

Combine both, then merge ranks. Recall neither gets alone.

Vector search misses literal IDs; keyword misses paraphrase. Run both.

05
Metadata filter — before, not after
🔐
Permission — access-group / tenant. Never retrieve what this user can't see.
🗓️
Freshness — date · version. Drop deprecated docs before they reach the model.
🏷️
Scope — source · type · sensitivity. Constrain to the right corpus.
06
Rerank — the quality lever

First-pass retrieval is fast but coarse. A cross-encoder reranker scores each candidate against the question and reorders — so the best evidence sits at the top of the context, not buried at rank 30.

Retrieve wide, then rank tight.

07
Context assembly

Build one clean evidence packet: de-duplicate, trim to the token budget, keep source IDs attached, order by relevance. Garbage-in still means garbage-out — a tidy packet is what makes the answer groundable and citable.

08
Grounded answer + citations

The model answers from the packet, not from memory — and every claim carries a marker back to its source chunk. Citations make the answer auditable: a reader can verify it, and "no evidence → say so" beats a confident guess.

grounded in evidence cite the source refuse when unsupported
09
Evaluation — measure retrieval AND generation, separately
LayerQuestion it answersMetrics
Retrieval qualityDid we fetch the right evidence?recall · precision · relevance · freshness
Answer qualityDid the model answer correctly using it?groundedness · correctness · citation accuracy · refusal quality

A fluent answer can hide failed retrieval. Score both before you trust the system — golden questions, expected sources, rubrics, regression tests.

10
Common failure
⚠️
top-k ≠ correct evidence — shipping the raw top chunks straight to the model assumes "near in vector space" means "answers the question." It often doesn't.
🎭
Fluent ≠ grounded — a confident paragraph from bad context still reads great. Without rerank, filtering, and eval you won't notice.
🧱
Skipping the filter — no permission/freshness gate leaks private or stale docs into the answer.
11
Where it lives in the AI SDLC
Architecture→ Build→ Evaluation→ Monitor

Design the pipeline up front, build each stage, gate the release on evals, then watch retrieval relevance & groundedness drift in production.

RAG isn't a one-time integration — it's an evaluated, monitored capability.

12
The takeaway
RAG is a quality pipeline, not a vector-DB call. Rewrite the question, search two ways, filter, rerank, ground, cite, and evaluate — every stage is a place to measure and improve.
Retrieve wide→ Rank tight→ Ground & cite→ Evaluate
