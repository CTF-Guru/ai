HOW AI AGENTS WORK

The agent loop — think, act, observe, repeat until done
CTF Farooq Ali
· AI SDLC course · explainer series
v2026.06
01
The problem — a bare LLM only talks

A model on its own is text in → text out. It can't open a file, run a test, search the web, or chase a multi-step goal. It describes the work; it can't do it.

02
What an agent is — brain + tools + memory

Wrap a model in a loop with tools and memory and it stops merely answering — it starts pursuing a goal.

03
The agent loop — think → act → observe → repeat until done

The whole behaviour is one small cycle run over and over: decide, do one thing, look at what happened, decide again. No giant master plan — just the next right step, on repeat.

04
Tools — what it can actually do

Each tool is one action the agent can take. The brain picks which to call and with what arguments; the result comes back as the next observation.

📄 read / write files ⌨️ run commands ✅ run tests 🔎 search the web 🌐 call an API 🗄️ query a database

A growing catalogue of these arrives in a standard way through MCP — one plug, many tools (see the MCP poster).

05
The ReAct pattern — reason + act, interleaved

The loop framing comes from ReAct (Yao et al., 2023): the model interleaves a reasoning trace with actions.

💭 "I should check X"→ 🔧 call tool→ 👁️ observe

Thoughts guide actions; observations refine the next thought. Reasoning keeps the tool calls on-target instead of blind.

06
Planning

For anything multi-step, the agent first breaks the goal into an ordered set of steps, then works them one by one.

goal→ steps→ do · check↺ re-plan

When reality differs from the plan — a test fails, a file is missing — it re-plans rather than barrelling on.

07
Memory

Short-term

The context window — the running transcript of thoughts, actions & results this task.

Long-term

Facts retrieved when needed — files, notes, a vector store (RAG).

Memory is how it carries what it learned in step 3 into step 9. (See the Agent Memory poster.)

08
An agentic step, on the wire — thought → action → observation → next thought
// THOUGHT — the model reasons about the next step
Thought: "The user asked for a conflict-free day. I should read the itinerary first."
// ACTION — it emits a tool call (name + arguments)
Action: { "tool": "read_file", "args": { "path": "itinerary.json" } }
// OBSERVATION — the harness runs the tool, feeds the result back
Observation: "2 events overlap 15:00–16:00 (Museum, Lunch)"
// …and the loop turns again, now better informed
Thought: "Conflict found. Next: propose moving Lunch to 13:00."

Thought, then one action, then the result — over and over. That repeating trace is the agent at work.

09
How far it goes — the autonomy ladder · you own intent + guardrails
10
Single vs multi-agent

One agent

A single loop handles the whole job end-to-end.

Orchestrator

A lead agent delegates to specialist sub-agents, then merges results.

Sub-agents coordinate over an agent-to-agent (A2A) protocol — the same plug-in idea, for agents talking to agents.

11
What nobody tells you
🪟
Context limits — the window fills; old steps fall out of view.
🧰
Tool sprawl — too many tools = more chances to mis-pick. Curate.
🔁
Runaway loops — it can spin without progress. Cap steps & cost.
🛟
Guardrails — human checkpoints, sandboxing, approvals before risky acts.
12
The takeaway
An agent doesn't make the model smarter — it gives it agency: the power to act, check its own work, and keep going until the job is done.
