01
What MCP is — the next-level architecture view

MCP standardizes how AI applications connect to external tools, data sources, prompts, and workflows.

One open protocol in the middle, so the model and the systems it touches stay cleanly separated. Build a connector once → every MCP-aware app can use it.

open standard JSON-RPC 2.0 model ⟷ systems boundary reusable connectors
02
The architecture flow — user request → app → client → server → systems → response
03
The three actors

🖥️ Host

The AI application the user talks to. Owns the conversation & the model.

🔌 Client

The connector inside the host that speaks MCP. One per server.

🗄️ Server

Exposes capabilities. A local process or a remote service.

Host holds Clients · each Client talks to one Server.

04
What a server exposes

🔧 Tools

An action the model can request.search_db · send_email

📄 Resources

Data the model can read.files · schemas · docs

💬 Prompts

A reusable task template the server offers./summarize

Three primitives — the whole mental model of MCP.

05
What flows through MCP
tools· resources· prompts→ results

The client discovers what a server offers, the model calls a tool or reads a resource, and a structured result flows back — request → response, every time.

06
Why it matters
🔗
Standard integration — one protocol instead of N×M bespoke connectors.
♻️
Reusable connectors — build a server once; any MCP-aware host plugs in.
🧱
Cleaner separation — a real boundary between the model and your systems.
07
Common trap — MCP is an integration pattern, not automatic safety
🔒
Permissions still need design — connecting a system doesn't decide what the model is allowed to do with it. Scope every tool.
✅
Validation is on you — arguments & results cross a trust boundary — validate inputs and outputs, don't assume them safe.
🪪
Identity is explicit — "who is acting" and under whose credentials is your call to make, not the protocol's.
🧾
Logging isn't free — no standard audit trail comes built in — log every call so actions are reviewable.

MCP connects systems. Permissions, validation, identity & logging are still yours to build.

08
A concrete picture — one host, three servers, three real systems
09
Builder takeaway
Design the server, not just the call. Treat every tool like a production API — scoped, validated, logged, revocable — and the architecture stays clean as you scale.
Expose a server→ Scope its tools→ Validate & log→ Reuse everywhere
10
Where it fits in the AI SDLC
Architecture→ Build→ Integration

MCP is decided in architecture (the boundary between model and systems), realized in build (servers + tools), and proven in integration — where permissions, validation, identity & logging are actually wired in.
