01
What release gates are — the checkpoints between a demo and production

Release gates are checkpoints that decide whether an AI feature is safe, useful, secure, and ready for more users. A demo that works once isn't a production system — each gate is a decision to promote the feature to the next ring, or send it back to fix.

🚦 A gate is a decision

Pass criteria → promote to more users. Fail → route back to fix, don't ship.

🛡️ Guardrails are different

Gates run before launch; guardrails run during every request in operation.

📈 Risk drops ring by ring

Prototype → eval → review → red team → pilot → monitor → scale. Exposure widens only as evidence grows.

02
The release pipeline — a feature card moves through the gates; a failed gate routes back to fix

Gates widen exposure only as evidence grows. A feature that clears every gate earns more users; one that fails any gate goes back — it never ships on hope.

03
Gate criteria — what each checkpoint checks
🎯
Quality threshold — eval scores clear the bar on a held-out test set, not a happy-path demo.
📎
Grounding / citation — answers trace to real sources; hallucination stays under the limit.
💉
Prompt-injection resilience — red-team suite can't make it ignore instructions or leak.
🗄️
Data-access review — what it can read and act on is scoped, least-privilege, signed off.
✅
Human-approval design — high-risk actions pause for a person — by design, not by luck.
🪙
Cost / latency budget — per-request spend and response time stay inside the agreed envelope.
↩️
Rollback plan — a tested way to disable, revert, or pause — ready before launch, not during the incident.
04
Runtime guardrails — what runs during operation

Gates clear the feature to ship. Guardrails protect every live request after that:

🧹
Input validation — sanitise and bound what reaches the model on each call.
🔎
Output checks — filter unsafe, ungrounded, or policy-violating responses before they reach users.
🔐
Tool permissions — scoped, revocable rights — the agent can only do what it's allowed.
🙋
Human approval — risky actions still pause for a person in the live flow.
🚨
Monitoring alerts — quality, cost, error, and abuse signals page someone when they drift.
05
Gates vs guardrails — before launch vs during operation
DimensionRelease gatesRuntime guardrails
WhenBefore launch / before each promotionDuring operation, every request
QuestionIs this safe to ship to more users?Is this single response / action safe?
Runs how oftenOnce per release decisionContinuously, per call
ExampleRed-team suite, eval threshold, rollback planInput validation, output filter, tool scope
If it failsRoute back — don't promoteBlock the call, alert, fall back

Two layers, not a choice. Gates decide what earns production; guardrails keep it safe once it's there.

06
The common failure
🩹
Leaning on runtime filters alone — teams skip the gates and hope an output filter catches everything at the edge.
🕳️
Runtime filters can't compensate — for missing evals, bad data access, or unsafe tool design. A filter can't un-leak a database the agent should never have reached, or invent a quality bar that was never measured.

The unsafe tool, the over-broad credential, the un-evaluated quality — those are decided upstream, at the gates. A guardrail is the last line, not the only one.

07
The takeaway
Use gates before launch and guardrails during operation. Gates decide what earns more users; guardrails keep each live request safe. Neither replaces the other.
Gate before launch→ Guardrail during operation→ Monitor & alert→ Fail → route back
08
Where it lives in the AI SDLC — Build → Test → Release → Monitor
Build→ Test→ Release→ Monitor

Build the feature · prove it with evals and red-team tests · gate the release · guard and watch it in production. Release is a decision, not an event.
