Did we retrieve right — and answer right? Two separate scorecards, one launch decision.

01
What RAG evaluation measures — retrieval quality AND answer quality, scored apart

RAG quality has two halves: did the system retrieve the right evidence, and did the model answer correctly using it. Score them separately — a confident, fluent answer can sit on top of completely failed retrieval, and a single end-to-end "looks good" number hides which half is broken.

retrieve evidence→ assemble context→ generate answer→ cite + check
02
Two scorecards — a question flows through retrieval eval, then answer eval, into the gate

Evidence is judged before the answer is. If retrieval fails, the answer score is meaningless.

03
Retrieval quality metrics
🎯
Recall — did we find the evidence the answer actually needs? Misses here cap every downstream score.
🧹
Precision — did we avoid dragging in irrelevant chunks that dilute the context?
🔗
Relevance — does the retrieved evidence match what the question is asking?
⏱️
Freshness — is the source current enough to still be true today?
📚
Coverage — did we include all the key sources, or only part of the picture?
04
Answer quality metrics
✅
Correctness — is the answer factually right for the question asked?
⚓
Groundedness — is every claim supported by the retrieved evidence, not invented?
🧩
Completeness — does it cover what was asked, without leaving gaps?
📎
Citation accuracy — do the cited sources actually say what the answer claims?
🚫
Refusal quality — when evidence is missing, does it decline instead of guessing?
05
The evaluation assets — what you build once, then run every release

📋 Golden questions

A fixed, representative set of real questions with known-good answers — your regression baseline.

📄 Expected sources

For each question, the documents that should be retrieved — this is how you score recall.

📏 Rubrics

Explicit pass/fail criteria per metric, so scoring is consistent across runs and graders.

👤 Human review

Sampled judgments on a slice of outputs — calibrates and audits the automated scores.

🔁 Regression tests

Re-run the golden set on every change; a passing build yesterday must still pass today.

📊 Scorecards

One retrieval card, one answer card — track both over time, not a single blended number.

06
The failure that hides
🎭
Fluency masks failure — a confident, well-written answer can sit on top of completely wrong or missing evidence.
🙈
End-to-end only — scoring just the final answer hides which half broke — retrieval or generation.
📉
Top-k ≠ correct — returning some chunks is not the same as returning the right evidence.

A fluent answer can hide failed retrieval.

Two numbers tell you where to look. One number doesn't.

Where it lives in the AI SDLC — evaluation is a gate, not an afterthought
09
The takeaway
RAG quality is two questions, not one. Evaluate retrieval and generation separately — score the evidence before you trust the answer, and gate the release on both.
Build golden set→ Score retrieval→ Score answer→ Gate · then monitor
