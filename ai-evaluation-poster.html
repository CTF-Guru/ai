01
What AI evaluation is — measuring readiness, not just correctness

AI evaluation measures whether an AI system is useful, safe, grounded, reliable, and ready for release. Deterministic code is right or wrong; an AI system is good enough, on a distribution of inputs — so you score it against suites, not single assertions.

Classic testing asks "does this function return the expected value?" Evaluation asks "does this system behave well across many real inputs, and is it safe to ship?"

02
The eval stack — the testing pyramid, evolved for AI

Unit + integration tests still apply — but they only cover the deterministic shell. The new layers test the model's behaviour: what it retrieves, which tools it picks, how it resists attack, and how it holds up live.

03
Why testing isn't enough

Non-deterministic

Same prompt, different output. Pass/fail becomes score over a set.temperature > 0

Open-ended

No single right answer — judge on a rubric, not equality.assertEquals ✗

New failures

Hallucination, injection, bad tool calls — unseen in code.silent & plausible

You can't unit-test your way to trust. You evaluate.

04
What an eval looks like
📋
Golden question sets — curated Q→expected-behaviour pairs you re-run every change.
🔗
Expected-source checks — did the answer cite the document it should have?
📊
Rubric scoring — grade correctness, grounding, tone on a scale (often LLM-as-judge + spot human).
🛡️
Prompt-injection tests — hostile inputs that try to override instructions.
🔁
Regression tests — yesterday's fixed bug must stay fixed.
💸
Cost / latency thresholds — fail the build if a turn gets too slow or too pricey.
05
Outputs run the gate — every candidate is graded: pass · review · fail

Same gate every run — promote what passes, escalate the uncertain, and route failures back before they reach users.

07
For agents: tools & safety
🧰
Tool-use evals — did it pick the right tool, with valid arguments, in the right order?
🚧
Safety / security evals — injection resilience, refusal quality, no unsafe action without approval.

More autonomy = more to evaluate before you hand over the keys.

08
The failure that ships broken AI
🎬
"It worked in the demo." One lucky run on one input proves nothing about the distribution real users bring.
🫥
Silent regressions — a prompt tweak or model swap quietly breaks cases that used to pass; no eval = no alarm.
📈
Vibes, not numbers — "feels better" can't gate a release. You need a score and a threshold.

A demo that works once is not a production-ready AI system.

09
Where it sits in the AI SDLC
Test→ Evaluate→ Release Gate→ Monitor

Evals run offline in CI (block the merge if scores drop) and become the release gate before more users, tools, or autonomy — then the same checks run live as production monitoring. Eval is not a phase; it's a loop.

10
The takeaway
Tests check the code around the model. Evals check the model itself. Build the eval suite before you scale users, tools, and autonomy — it's the seatbelt, not the trophy.
Write evals first→ Gate on scores→ Then scale→ Monitor live
